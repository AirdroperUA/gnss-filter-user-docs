# Огляд пристрою

> Де купити плату: [GPS Spoofing Filter](https://airdroper.org/product/gps-spoofing-filter/)

Цей документ описує, як STM32-фільтр працює між GNSS-приймачем і польотним контролером (FC).

## Терміни (короткий словник)

- **DR (dead-reckoning)**: оцінка позиції БПЛА за інерціальними даними (IMU), курсом і швидкістю, коли GNSS не довіряють. Дає безперервність навігації, але поступово накопичує похибку.
- **DR0**: штатний режим. Дані GNSS передаються на GPS UART FC.
- **DR1**: захисний режим. Живе передавання GNSS блокується — FC GPS UART отримує тишу, щоб підозрілі дані не потрапляли у FC.
- **FC**: польотний контролер (ArduPilot).
- **GNSS**: супутниковий навігаційний приймач (наприклад u-blox класу M8/M9/M10/F9/F10 або приймачі класу UM980/UM981).
- **GPS**: у цій документації шлях GNSS-входу і GPS UART на FC.
- **UART**: послідовний інтерфейс із лініями TX/RX.
- **MAVLink**: протокол телеметрії/керування між STM32 і FC.
- **GCS**: наземна станція керування (Mission Planner, QGroundControl).
- **UBX**: бінарний протокол GNSS від u-blox.
- **NMEA**: текстовий GNSS-протокол, який використовують багато приймачів.
- **EKF**: розширений фільтр Калмана всередині FC.

## Сумісність і безпека

- Потрібен **ArduPilot 4.6.1 або новіший**.
- Режим приймача задається параметром `GNSS_TYPE`:
  - `GNSS_TYPE=0`: режим u-blox/UBX.
  - `GNSS_TYPE=1`: режим UM980/UM981 (NMEA).
- Для `GNSS_TYPE=1` фільтр очікує один фізичний потік UM980:
  - `COM1` -> STM32 `A2/A3`
  - STM32 читає spoofing/SNR-дані з цього потоку
  - STM32 пересилає цей самий потік на GPS UART FC
- `GNSS_TYPE` зберігається одразу, але застосовується тільки після перезавантаження STM32.
- Перші польоти виконуйте на недорогій платформі, яку легко відновити (наприклад, 10-дюймовий FPV-квадрокоптер з ArduPilot або невелике крило типу Reptile Dragon V2). Лише після перевірок ставте систему на дорогий БПЛА.

## 1) Призначення

Фільтр встановлюється між GNSS і FC та виконує три основні задачі:

1. Парсить GNSS-дані і оцінює якість/аномалії.
2. Керує станом DR (DR0/DR1) за логікою захистів.
3. Керує передаванням GNSS на GPS UART FC (у DR1 блокується).

## 2) Архітектура системи

│              │                │                   │                │              │
│  u-blox або  │                │  ┌─────────────┐  │   UART 460800  │  ArduPilot   │
│  UM980       │                │  │ Guard Logic │  │───────────────►│  GPS вхід    │
│              │                │  │ DR0 / DR1   │  │  A11/A12       │              │

                                │                   │

                                └───────┬───────────┘
                                        │
                                   (DR0→DR1 імпульс)

## 3) Шляхи даних

Є три UART-з'єднання:

| Лінк | Піни STM32 | Бод | Призначення |
|------|-----------|-----|-------------|
| GNSS ↔ STM32 | A2 (TX) / A3 (RX) | 460800 | Вхід приймача для перевірок якості |
| FC MAVLink ↔ STM32 | A9 (TX) / A10 (RX) | 115200 | Статус, команди, тюнінг |
| FC GPS ← STM32 | A11 (TX) / A12 (RX) | 460800 | Передавання GNSS на FC |

Для UM980/UM981 достатньо одного фізичного UART від приймача. STM32 споживає цей єдиний GNSS-потік і дзеркалить його на FC GPS UART.

У DR0 дані GNSS передаються на GPS UART FC.

У DR1 живе передавання GNSS блокується — FC GPS UART отримує тишу (жодних GNSS-даних).

## 4) DR0 та DR1

### DR0 (штатний режим)

- Передавання GNSS увімкнене.
- Захисти постійно контролюють якість.

### DR1 (захисний режим)

- Живе передавання GNSS вимкнене.
- FC GPS UART отримує тишу (жодних GNSS-даних).
- Пін `B5` видає високий імпульс приблизно на 3 секунди при кожному переході DR0 -> DR1.

## 5) Що може ввімкнути DR1 (приклади)

### Приклад A: no-fix або мало супутників

- Фікс GNSS стає невалідним або кількість супутників падає нижче 5.
- Фільтр переходить у DR1 одразу (поза стартовою затримкою).

### Приклад B: стрибок позиції

- Різкий великий стрибок координат або нереалістична швидкість між фіксами.
- Перевищено `SP_ABS_M` або `SP_JMP_MPS`.

### Приклад C: аномалія SNR (близький глушник/спуфер)

- Висока кількість супутників при аномально вузькому розкиді SNR.
- Умова тримається довше `SNR_HOLDMS` при `SNR_EN=1`.

### Приклад D: аномалія висоти

- Великий разовий стрибок висоти, надмірна швидкість зміни висоти або стійке розходження GNSS проти баро.

### Приклад E: EKF flags trip

- EKF на FC довго показує невалідний горизонтальний стан (не менше `EKF_TRIPMS`).

## 6) Повернення з DR1 (rejoin)

Фільтр повертається у DR0 тільки після виконання налаштованих умов якості й часу:

- якість (`RJ_MIN_SATS`, `RJ_MAX_HD`),
- вікно стабільності (`RJ_STAB_MS`),
- блокувальне вікно DR1 (`DR_LOCK_MS`),
- опційна EKF-умова (`RJ_REQEKF`, `EKF_OKRJMS`).

Після виконання умов може запускатися blend (`BLEND_MS`), потім відновлюється DR0.

## 7) Ключові робочі примітки

- `FCGPS_FWD=1` використовуйте тільки для діагностики, бо він обходить блокування DR1 / режим NO_FIX.
- `BOOT_DLYMS` додає стартову затримку DR-тригерів і зменшує хибні спрацювання після вмикання.
- Карта GCS може показувати GNSS-стрибки під час спуфінгу; основне джерело істини - стан DR і логи фільтра.
- Перед польовими змінами зберігайте базовий профіль параметрів.

## 8) Сумісність із системами РЕБ

GPS Spoofing Filter захищає від атак спуфінгу та глушіння GPS від будь-якої системи радіоелектронної боротьби (РЕБ). Алгоритми захисту фільтра виявляють аномалії в супутникових сигналах незалежно від джерела РЕБ — виявлення базується на фізиці, а не на сигнатурах.

> **Примітка:** Фільтр виявляє спуфінг та глушіння за аномаліями сигналу (стрибки позиції, патерни SNR, розходження висоти, зменшення кількості супутників), а не за ідентифікацією конкретного обладнання РЕБ. Це означає, що він працює проти **будь-якої** системи РЕБ, яка впливає на сигнали GNSS — включаючи системи, не перелічені тут, та майбутні системи, які ще не розгорнуті.

## 9) Вибір приймача: UM980/UM981 проти u-blox (для цього фільтра)

UM980/UM981 часто є кращим варіантом для довгих anti-spoof сценаріїв у цьому проєкті, головно через:

- стабільніше утримання супутників у складному RF-середовищі;
- зазвичай кращу поведінку після тривалих завад у польових умовах (швидше повернення до робочого фікса);
- роботу в NMEA-режимі (`GNSS_TYPE=1`) без залежності від UBX-переналаштування;
- високу доступність супутників у відкритому небі, що покращує якість guard/rejoin рішень.

u-blox також підтримується і працює штатно при правильному налаштуванні (`GNSS_TYPE=0`).  
Фінальний вибір робіть за результатами власних польотних тестів у вашому регіоні та умовах завад.
