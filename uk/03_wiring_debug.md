# Діагностика підключення

> Де купити плату: [GPS Spoofing Filter](https://airdroper.org/product/gps-spoofing-filter/)

Використовуйте цей документ, якщо GNSS не визначається, DR1 постійно активний, або не працює MAVLink-тюнінг.

## 1) Шлях GNSS (STM32 <-> GNSS)

Симптоми:
- `data=... fix=9999ms nav=9999ms ...` тримається або `fix` / `nav` постійно зростають.
- Супутники можуть бути присутні, але фікс усе одно невалідний.
- DR1 не вимикається через no-fix/low-sat тригер.

Перевірки:
1. **Живлення**: у GNSS є стабільне живлення та спільна земля зі STM32/FC.
2. **UART-піни**: GNSS TX -> STM32 `A3`; GNSS RX -> STM32 `A2`.
3. **Режим приймача**:
   - для u-blox: `GNSS_TYPE=0`;
   - для UM980/UM981/UM982 NMEA: `GNSS_TYPE=1`.
   - Для UM980/UM981/UM982 використовуйте лише один фізичний порт приймача: `COM1` -> STM32 `A2/A3`.
     Фільтр читає spoofing/SNR-дані з цього потоку і пересилає той самий потік на FC GPS UART.
4. **Випадок кастомного u-blox**:
   - Якщо ваш u-blox не приймає автоконфіг фільтра, задайте `UBX_BAUD` відповідно до baud приймача.
   - `UBX_BAUD=0` залишає автоконфіг увімкненим (за замовчуванням).
   - Якщо `UBX_BAUD>0`, фільтр пропускає повний шлях UBX-автоконфігу. Він усе одно робить best-effort запит на `NAV-SAT`, але `SNR=NA` все ще є нормальним, якщо приймач ігнорує цей запит.
   - Те саме правило ручного baud стосується і gateway-модулів з проміжним MCU (наприклад Quadro GPS, UNA3, UNA4-SFE або подібних dual-F9P продуктів). Задавайте точний вихідний `UBX_BAUD`; не покладайтеся на autobaud фільтра.
5. **Перезавантаження після зміни режиму/baud**: `GNSS_TYPE` і `UBX_BAUD` застосовуються лише після reboot STM32.
6. **Умови прийому**: для реальної перевірки GNSS потрібен відкритий огляд неба.

Швидка перевірка:
- У логах `fix` і `nav` мають бути малими, а `SATS` повинно відповідати реальному стану приймача.
- Якщо `nav` довго росте, фільтр не отримує валідних GNSS nav-data.
- Якщо `fix` довго росте, фільтр не отримує валідних позиційних фіксів.

## 2) Шлях FC GPS (STM32 <-> FC GPS UART)

Симптоми:
- FC показує **No GPS** або **No GPS config data**, хоча фільтр бачить GNSS-трафік.

Перевірки:
1. **UART-піни**: STM32 `A11/A12` до FC GPS UART, TX/RX перехресно.
2. **Протокол/baud на FC**:
   - типовий варіант для u-blox: GPS protocol + 460800 baud;
   - для UM980/UM981/UM982 протокол GPS на FC має відповідати виходу вашого приймача.
   - Якщо UM980 налаштовано як один змішаний потік `COM1`, FC має відповідати саме цьому пересланому потоку.
3. **Стан DR**: у DR1 живе пересилання заблоковане за дизайном — FC GPS UART отримує тишу.
4. **Діагностичний обхід**: тимчасово встановіть `FCGPS_FWD=1` для перевірки шляху FC GPS, потім поверніть `0`.
5. **RC AUX GPS disable**: якщо у FC/RC налаштовано вимкнення GPS через AUX, переконайтесь, що цей перемикач не активний.

Швидка перевірка:
- При здоровому GNSS і `FCGPS_FWD=1` FC має бачити супутники/фікс.
- Після перевірки поверніть `FCGPS_FWD=0` для штатного anti-spoof режиму.

## 3) Шлях MAVLink (STM32 <-> FC telemetry)

Симптоми:
- Mission Planner не читає/не записує параметри STM32.
- Немає статус-повідомлень фільтра у GCS.

Перевірки:
1. **UART-піни**: STM32 `A9/A10` до телеметрійного UART FC, TX/RX перехресно.
2. **Протокол**: на FC MAVLink2, 115200 бод.
3. **ID системи**: стандартно `SYSID=42`, `COMPID=191`.
4. **Один власник COM-порту**: закрийте інші серійні утиліти під час роботи в Mission Planner.

Швидка перевірка:
- У Mission Planner:
  1. `Config/Tuning` -> `Full Parameter List`
  2. виберіть STM32 (`SYSID 42`)
  3. натисніть `Refresh Params` і перевірте, що параметри зчитуються.

## 4) Типові помилки

- TX/RX не перехрещені.
- Переплутані GNSS UART і FC GPS UART.
- Немає спільної землі.
- Змінено `GNSS_TYPE`, але STM32 не перезавантажено.
- Змінено `UBX_BAUD`, але STM32 не перезавантажено.
- Перевірка відновлення DR одразу після старту, поки активна затримка `BOOT_DLYMS`.

## 5) Проблеми запису параметрів у Mission Planner

Якщо читання/запис параметрів нестабільний:

1. Залиште тільки один GCS на COM-порту.
2. У `Full Parameter List` натисніть `Refresh Params` перед редагуванням.
3. Змініть значення, натисніть `Write Params`, потім `Refresh Params`.
4. Якщо значення не збереглося, повторіть `Write Params`.
5. На завантаженому телеметричному лінку 1-2 спроби запису є нормальними; більше 2 вказує на перевантажений лінк.
6. Якщо лінк тимчасово просів, зачекайте до 30-45 секунд і оновіть ще раз.
7. Перезавантажуйте STM32 лише для `GNSS_TYPE` або `UBX_BAUD`, або якщо значення все ще не застосовується.
